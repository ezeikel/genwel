# Genwel Audit & Roadmap

_Generated 2026-06-22. Source: 8-agent audit workflow (5 internal code audits + 3 external research: AppKittie market intel, Mobbin UX patterns, Context7/web version research) + direct DB inspection of both Neon branches._

## Current State

Genwel is a Next.js 16 monorepo with a polished, near-complete marketing site and a structurally complete authenticated dashboard, but it is NOT a working MVP. The core product loop is broken: a user can sign in (next-auth beta.28 + Resend magic links), connect a bank via TrueLayer (sandbox mock only, john/doe), and see 5 account balances — but no transactions ever appear. I verified that neither app/dashboard/page.tsx nor app/dashboard/transactions/page.tsx call any sync function; transaction sync lives only inline in actions/banking.ts getTransactions() and is never triggered from the main flow. AI categorization (the entire value prop) is therefore never run, so aiCategory is always null, which means budgets show 0 progress and insights are empty (getBudgetProgress groups by aiCategory). The fix exists but is stranded on the wip-transaction-sync-categorization branch (apps/web/lib/banking/sync.ts + categorize.ts, confirmed present), abandoned because awaiting those I/O-heavy calls in a server component froze the page for 5-10s. On the marketing side the site looks done but is functionally hollow: the waitlist form has no backend (no /api/waitlist endpoint exists — confirmed), legal pages (privacy/terms) are 404 placeholders, and CTAs/download buttons are dead. Security has real blockers for a finance app: TrueLayer access/refresh tokens are stored plaintext in Postgres, and the OAuth callback trusts a userId parsed from an unsigned state string instead of the session. NOTE: one audit claim is overstated — .env.local is gitignored and NOT tracked in git (verified), so secrets are exposed on local disk only, not committed history; still worth rotating since they appeared in audit output. The mobile app does not exist (empty placeholder, no package.json). Stack is modern but several "latest" premises in the brief are stale: AI SDK latest is v6 (repo on v4, a two-major jump), and the next-auth deleteSession/P2025 fix is still unmerged so the auth.ts try-catch must stay.

## Top Blockers

- 🔴 Transaction sync never runs in the main flow: connect bank shows balances but transaction list is permanently empty. Confirmed no sync/getTransactions call in app/dashboard/page.tsx or transactions/page.tsx. The whole product loop is dead until this is wired.
- 🔴 AI categorization never triggers, so aiCategory is always null — budgets show 0 progress and insights are empty because getBudgetProgress() and insights both key off aiCategory. This is the headline feature and it is inaccessible.
- 🔴 The fix is stranded on the wip-transaction-sync-categorization branch (lib/banking/sync.ts + categorize.ts confirmed present) and was abandoned because awaiting sync+categorize in a server component froze the page 5-10s. Needs a non-blocking architecture (background/deferred sync, not page-blocking await) before it can land.
- 🔴 Waitlist form is non-functional — no /api/waitlist endpoint exists (verified); emails are dropped. The marketing site's only conversion mechanism collects nothing.
- 🔴 Security blocker for a finance app: TrueLayer access/refresh tokens stored plaintext in Postgres (BankConnection.accessToken/refreshToken). Must be encrypted at rest before handling any real bank data.
- 🔴 Security blocker: banking OAuth callback trusts userId from an unsigned state string (state.split(':')) and never verifies it against the authenticated session — account-linking CSRF / hijack risk. Must derive userId from auth() and HMAC-sign or session-store the state.
- 🔴 Gemini free-tier rate limit (5 req/min) + Promise.all parallel image evaluation + zero retry logic will reliably 429 during blog generation and batch categorization. No backoff anywhere in the AI layer.
- 🔴 next.config.ts sets typescript.ignoreBuildErrors:true — type safety is fully disabled, masking real bugs across the app before a finance launch.

## MVP Path (web → live)

1. Fix the data loop (highest priority): create lib/banking/sync.ts (port from wip branch) as a plain async util, but call it non-blocking. Simplest reliable approach for MVP: sync the last 30 days inside api/banking/callback/route.ts right after account sync (so first connect seeds data), plus a lightweight 'Sync now' / background revalidate path on the dashboard. Do NOT await a 90-day full sync in the server component render.
1. Wire AI categorization off the page-render path: port lib/banking/categorize.ts (with merchant caching), but trigger it as a best-effort background/deferred job after sync, or behind an explicit 'Categorize' button for MVP. Add a category->aiCategory fallback heuristic so budgets/insights show *something* even before AI runs.
1. Add retry-with-backoff to all AI calls (create lib/ai/retry-utils.ts) and change image-evaluation.ts from Promise.all to sequential/concurrency-limited (1 for Gemini) to stop 429s. This also unblocks reliable categorization.
1. Encrypt TrueLayer tokens at rest (field-level encryption with a key from env) and migrate any existing rows. Non-negotiable before real (non-sandbox) bank data.
1. Harden the banking callback: derive userId from auth() session, HMAC-sign the state with NEXTAUTH_SECRET (or store state server-side with short TTL) and constant-time verify on callback.
1. Build /api/waitlist (POST) wired to Resend + persist to DB, emit posthog.capture('waitlist_signup'), and add UI success/error states to HeroSection/Footer/launch forms.
1. Ship legally-required pages: /legal/privacy and /legal/terms (and cookie notice). Wire dead CTAs to /waitlist or /signin. Resolve the alpha-vs-coming-soon messaging inconsistency to one clear status.
1. Add rate limiting to the magic-link sign-in flow (per-IP/per-email) and validate CRON_SECRET on /api/blog/generate.
1. Re-enable TypeScript checking: remove typescript.ignoreBuildErrors and fix the v0-component type errors. Add HSTS headers, set Sentry sendDefaultPii:false, and sanitize PostHog log context.
1. Manually QA the full loop end-to-end (connect mock bank -> transactions appear -> categories populate -> budget progress + insights render), then deploy apps/web to Vercel (link in apps/web, pull env) and verify production build passes with Turbopack.
1. Bump TrueLayer to live/production credentials and complete TrueLayer production onboarding (separate from sandbox) once the encryption + callback fixes are in.

### Genwel Authenticated Dashboard + Open Banking (TrueLayer)
The Genwel authenticated dashboard provides a solid UI structure for viewing accounts, transactions, budgets, and insights, with TrueLayer Open Banking integration for mock bank connectivity. However, the implementation has a critical architectural gap: transaction sync and AI categorization are missing from the main flow—transactions sync only via inline server-action logic in banking.ts when explicitly requested, never automatically. The stranded wip-transaction-sync-categorization branch contains the necessary utilities (sync.ts, categorize.ts) as plain library functions rather than server actions, designed to be called from server components for automatic syncing, but was abandoned due to page-freeze issues from awaited sync/categorize calls. The MVP lacks end-to-end transaction flow and AI-powered categorization is inaccessible until manual action is triggered.

- 🟠 **Transaction Sync Architecture: Inline vs. Utility Functions** (high)  
  Currently, transaction sync logic lives inline in actions/banking.ts getTransactions() function (lines 197-260), triggered only when explicitly called from pages. It fetches and upserts the last 30 days of transactions on-demand but is never called from the main dashboard flow. The wip-transaction-sync-categorization branch provides a cleaner sync.ts utility (90 lines) as a pure function (not a server action), fetching 90 days retroactively, designed to be awaited from server components like DashboardPage. This creates duplication: the action version is production code, the branch version is more maintainable but was abandoned.  
  → _Move transaction sync to a dedicated lib/banking/sync.ts utility function similar to the wip branch (pure async function, not server action). Call it unconditionally from DashboardPage, TransactionsPage, and BudgetsPage at the top, before querying the DB. This ensures transactions are always fresh and AI categorization has data to work with. The key fix: call syncUserTransactions(userId) before any DB queries, as the wip branch does._
- 🔴 **AI Categorization: Never Triggered on Main Branch** (blocker)  
  The AI categorization system exists in lib/ai/categorization.ts and actions/ai-budgets.ts but is never called automatically. The Dashboard page doesn't trigger categorization. The wip branch adds categorizeUserTransactions() utility in lib/banking/categorize.ts with merchant caching (93 lines) and calls it from dashboard pages (line 28-29 of wip's page.tsx). BudgetProgressList and insights components query aiCategory field, but if transactions aren't categorized, the budget and insights pages show no data. This is the 'sticky point': categories are null until the user manually clicks a button (if such a button exists on the frontend).  
  → _Add categorizeUserTransactions() utility to lib/banking/categorize.ts as a plain function. Call it automatically after syncUserTransactions() in dashboard/page.tsx, transactions/page.tsx, and budgets/page.tsx. For MVP, make this a best-effort call that silently fails if insufficient data exists. Alternatively, add a UI button 'Categorize Transactions' on the dashboard that calls the categorizeTransactions() server action, making categorization a conscious user action. Current state: budgets show 0 progress and insights are empty because aiCategory is always null._
- 🟠 **Page Freeze Issue in wip Branch: Awaited Sync/Categorize** (high)  
  The wip branch calls `await syncUserTransactions(userId)` and presumably `await categorizeUserTransactions(userId)` from server components (DashboardPage, etc.). These are I/O-heavy operations: sync fetches from TrueLayer for all accounts (10+ API calls for 5 accounts), categorize makes 1-2 AI API calls per batch of 20 transactions. Awaiting in a server component blocks the entire page render, causing user-facing freezing. The wip branch was abandoned because the page took 5-10s to load.  
  → _For MVP, implement sync as a time-boxed background task: call syncUserTransactions() in a background Job/queue (e.g., node-cron scheduled job, or defer via fetch in the API route) rather than awaiting it in the page. Return stale data immediately to the user. Show a 'Syncing...' badge. For categorization, make it a manual trigger (button) or async background job, not a page-blocking operation. Alternative: reduce TrueLayer fetch scope (e.g., sync only the last 7 days, cache heavily) to keep latency under 1-2s so awaiting is acceptable._
- 🔴 **Mock Bank Connects (5 Accounts) but Transactions Never Appear** (blocker)  
  The flow: User connects bank → callback route (api/banking/callback/route.ts) creates BankConnection and syncs accounts/balances immediately → dashboard shows 5 accounts with balances. BUT no getTransactions() call happens at this stage. Dashboard doesn't call getTransactions() at render time. TransactionsPage would call getTransactions(), but users rarely navigate there first. Result: transaction table is empty, budget sync can't run, insights are empty. The user sees a dashboard with account balances but no financial activity.  
  → _Call syncUserTransactions(userId) in DashboardPage.tsx before rendering (at the very top of the function, after auth checks, before any DB queries). This ensures at least the last 90 days of transactions are populated on first page load. Alternatively, sync on-demand in the callback route itself: add a TrueLayer.getTransactions() call after account sync in api/banking/callback/route.ts, fetching the last 30 days._
- 🔵 **No Merchant Caching in Production Code** (low)  
  actions/ai-budgets.ts categorizeTransactions() implements merchant caching (lines 45-59: distinct query on merchantName, map-based lookup). This is efficient for batch categorization. However, this logic is only used when categorizeTransactions() is explicitly called, which doesn't happen automatically. The wip branch's categorize.ts duplicates this logic. Both implementations are correct but isolated.  
  → _Consolidate: move merchant caching to lib/banking/categorize.ts once (as the wip branch suggests), and have both the server action and automatic-call path use the same utility. This prevents divergence and ensures consistency._
- 🔴 **Dashboard UX: Components Render Correctly, But No Data Flow** (blocker)  
  DashboardPage.tsx renders BalanceCard, TransactionList, BudgetStatusSummary, and EmptyState components correctly. TransactionList and SpendingChart exist. BudgetProgressList, InsightsList exist. However, data is missing at the source: (1) TransactionList receives 0 transactions because transactions aren't synced. (2) BudgetStatusSummary queries aiCategory, which is always null, so no budget progress can be calculated. (3) InsightsList is empty because aiCategory is null. The UX flow is complete structurally but broken functionally.  
  → _Fix the upstream data flow: sync transactions → categorize → render. Once these steps work, the UX will display correctly. Test the full flow: connect bank, sync last 90 days, categorize, view dashboard. BudgetStatusSummary should show top 3 categories, transaction list should show 10 recent items._
- 🟠 **Budget System: Depends on aiCategory, Not 'category' Field** (high)  
  getBudgetProgress() in actions/budgets.ts (lines 153-162) groups transactions by aiCategory, not category. The category field is populated from TrueLayer's mapTransactionCategory() (e.g., 'Shopping', 'Bills'). aiCategory is the AI-categorized field (e.g., SHOPPING, GROCERIES). Budget lines are defined in SpendingCategory enum (SHOPPING, GROCERIES, etc.). So the budget system is intentionally AI-first: it ignores TrueLayer's basic categories and requires AI categorization to function. This is a design choice but creates a critical dependency.  
  → _Document this clearly in code comments. For MVP, consider adding a fallback: if aiCategory is null, map category → aiCategory using a simple heuristic (e.g., 'Shopping' → 'SHOPPING'). This unblocks budget progress even without AI. Or, on user setup, auto-run categorizeTransactions() to ensure all transactions are categorized before showing the budgets page._
- 🟡 **No Retry or Error Handling for TrueLayer Sync** (medium)  
  Both actions/banking.ts getTransactions() and the wip sync.ts catch errors silently (console.error, then continue). If TrueLayer API is slow or times out, users get a broken experience without feedback. No retry logic, no queue, no webhook fallback.  
  → _Add exponential backoff retry logic to getTrueLayerTransactions() calls (3 retries, 1s / 2s / 4s delays). If all retries fail, throw an error that propagates to the UI (show a toast: 'Failed to sync transactions; please try again'). For production, implement a background queue (Bull, Inngest) to retry failed syncs asynchronously._
- 🟡 **Token Refresh Logic is Duplicated and Fragile** (medium)  
  Token refresh logic is duplicated across actions/banking.ts in getAccounts() (lines 55-82) and getTransactions() (lines 202-218). Both check if token is expired or expiring within 1 hour, then refresh. No deduplication. If refresh fails, the function logs and continues without the updated token, leading to 401 errors on the next TrueLayer call.  
  → _Extract token refresh to a utility: lib/truelayer/token-refresh.ts with a refreshIfNeeded(connection) function. Call it once before any TrueLayer API call. Ensure the connection.accessToken is updated in memory so subsequent calls use the fresh token. Add tests to verify refresh is only called once per connection per session._
- 🔵 **No Data Freshness Indicators in UI** (low)  
  BalanceCard, AccountCard, and TransactionList render balances and transactions without showing 'Last synced at' timestamps. User has no way to know if data is stale. The balanceUpdatedAt field exists in the schema but isn't displayed.  
  → _Add a small 'Last updated X minutes ago' label below account balances and transaction lists. Show a 'Syncing...' spinner during sync. This gives users confidence in data freshness and communicates when background updates are happening._
- 🔵 **Connection Disconnect Flow: No UI Confirmation or Cascade Cleanup** (low)  
  disconnectBank(connectionId) in actions/banking.ts (lines 325-358) exists but there's no UI button to trigger it. AccountCard component doesn't expose a disconnect option. Deleting a connection cascades to bankAccounts and transactions (Prisma CASCADE), but users can't trigger this flow.  
  → _Add a 'Disconnect' button to AccountCard component (inside a dropdown menu). On click, show a confirmation dialog: 'This will delete all transactions and accounts from this connection. Continue?' On confirm, call disconnectBank(). Revalidate dashboard._
- 🟡 **API Route /api/banking/callback Not Re-syncing After Initial Connect** (medium)  
  api/banking/callback/route.ts (lines 71-97) syncs accounts and balances on first connection but doesn't fetch transactions.
